# 项目路线图（个人开发成熟案例）

## 1. 目标定位
本路线图面向“长期运营的个人项目”，而不是一次性交付。核心目标是：
1. 先打通核心业务闭环。
2. 再提升稳定性与可观测性。
3. 最后强化平台化能力与可扩展性。

## 2. 分阶段规划
## 阶段一：MVP（第1~4周）
交付重点：
1. 事件接入、规则判定、告警闭环。
2. MySQL + Redis + InfluxDB 基础读写打通。
3. 管理端基础查询与 WebSocket 推送。

验收门槛：
1. 端到端告警可稳定触发。
2. 关键接口具备鉴权与参数校验。
3. 有基础部署文档和启动脚本。

## 阶段二：Beta（第5~8周）
交付重点：
1. 缓存策略优化与流式消费重试机制。
2. 趋势统计、排行榜、报表导出。
3. 监控指标、结构化日志、核心告警策略。

验收门槛：
1. `P95` 告警延迟达标。
2. 压测达到 `100~300 events/s`。
3. 具备故障恢复与死信处理能力。

## 阶段三：Stable（第9~12周）
交付重点：
1. 灰度发布与回滚流程。
2. 规则版本管理与灰度生效。
3. 备份恢复演练与安全基线加固。

验收门槛：
1. 72小时稳定性测试通过。
2. 关键运维操作具备 Runbook。
3. 核心安全测试项闭环完成。

## 3. 演进路径建议
1. 不急于微服务化，先守住模块边界。
2. 以“监控数据”驱动优化，不做盲目重构。
3. 优先解决瓶颈模块：先写入链路，再查询链路。
4. 保持每两周一个可发布版本。

## 4. 风险与应对
| 风险 | 影响 | 应对策略 |
|---|---|---|
| 需求膨胀 | 进度失控 | 固定迭代目标，明确延期池 |
| 流量突增 | 告警延迟上升 | 扩消费者、批量写入、短期限流 |
| 数据膨胀 | 成本与性能压力 | 分层保留、聚合归档、冷热分离 |
| 误报偏高 | 业务可信度下降 | 调参回放、规则灰度、误报闭环 |

## 5. 成熟度检查清单
1. 功能成熟：告警流程、统计、权限、审计完整。
2. 工程成熟：配置治理、环境隔离、自动化测试。
3. 运维成熟：监控、告警、备份恢复、故障手册。
4. 安全成熟：鉴权、脱敏、审计、密钥管理。
